# ID Disambiguation in Lexical-Loro Collaboration

## Problem Summary

There was confusion between two different types of IDs in the collaborative system:

1. **WebSocket Client ID** - Generated by the server for each WebSocket connection
2. **Loro Peer ID** - Generated by Loro's CRDT system for each document peer

This confusion led to ambiguous naming and potential bugs where these IDs were mixed up.

## Solution Implemented

### Server-Side Changes (`lexical_loro/serverv2.py`)

1. **Clear Documentation**: Added comments explaining the two ID types:
   ```python
   # Two types of IDs in this system:
   # 1. WebSocket Client ID: Generated by server for each WebSocket connection (self.clients keys)
   # 2. Loro Peer ID: Generated by Loro CRDT system for each document peer (doc.peerIdStr)
   # These are mapped together: WebSocket Client ID -> Loro Peer ID
   ```

2. **Parameter Naming**: Updated method parameters to be explicit:
   - `client_id` → `websocket_client_id` in method parameters
   - `exclude_client` → `exclude_websocket_client` in broadcast methods

3. **Response Format**: Updated peer data structure to clearly distinguish IDs:
   ```python
   peer_list.append({
       "id": loro_peer_id,  # Loro peer ID (CRDT concept)
       "websocketClientId": websocket_client_id,  # WebSocket client ID (connection concept)
       "displayId": ...,
       "isCurrentUser": False  # Will be set by client based on WebSocket client ID match
   })
   ```

### Client-Side Changes (`src/LoroCollaborationPluginV2.tsx`)

1. **Interface Updates**: Renamed callback to be explicit:
   ```typescript
   // Old
   onPeerIdChange?: (peerId: string) => void;
   
   // New
   onWebSocketClientIdChange?: (websocketClientId: string) => void;
   ```

2. **Documentation**: Added clear comments explaining the two ID types:
   ```typescript
   // Note: Two types of IDs in the system:
   // 1. WebSocket Client ID: Generated by server for each WebSocket connection (provider.clientId)
   // 2. Loro Peer ID: Generated by Loro CRDT system for each document peer (peer.id)
   ```

3. **Peer Processing**: Updated peer handling to use correct field names:
   ```typescript
   // Old
   peer.clientId === provider.clientId
   
   // New
   peer.websocketClientId === provider.clientId
   ```

### Example Component Changes (`src/examples/LexicalCollaborativeEditorV2.tsx`)

Updated the component to use the new callback name:
```typescript
// Old
onPeerIdChange={handlePeerIdChange}

// New
onWebSocketClientIdChange={handlePeerIdChange}
```

## ID Usage Guidelines

### When to Use WebSocket Client ID
- **Connection Management**: Identifying which WebSocket connection sent/receives messages
- **Session Tracking**: Determining if a peer is the current user's session
- **Server Operations**: Broadcasting, disconnection cleanup, etc.

### When to Use Loro Peer ID
- **CRDT Operations**: Document updates, collaboration state
- **User Display**: Showing peer names/avatars in UI
- **Awareness Data**: Cursor positions, selections, user presence

## Benefits

1. **Clarity**: No more confusion about which ID is being used where
2. **Maintainability**: Code is self-documenting with explicit naming
3. **Debugging**: Easier to trace issues between connection and collaboration layers
4. **Type Safety**: Better TypeScript types prevent mixing up ID types

## Migration Notes

If extending this system:
- Always specify whether you're working with WebSocket Client IDs or Loro Peer IDs
- Use the mapping `client_loro_peer_ids` to convert between them when needed
- Keep the server response format consistent with the new field names
